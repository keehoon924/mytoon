# improve-codebase-architecture

## 목적
코드베이스를 더 깊은 모듈로 진화시켜 복잡도를 외부에서 숨긴다.
이슈 5 개 이상 누적 시점에 한 번씩 돌리는 정비 스킬.

## 배경 개념

### 깊은 모듈 vs 얕은 모듈 (Ousterhout)
- **깊은 모듈**: 인터페이스 작음 + 내부 구현 풍부 → 호출자가 신경 쓸 게 적다.
- **얕은 모듈**: 인터페이스가 구현만큼 크다 → 추상화 가치 없음. 그냥 인라인이 낫다.

### 삭제 테스트 (deletion test)
"이 모듈/함수/추상화를 지우고 호출부에 인라인하면 더 단순해지는가?"
- YES → 얕은 모듈. 인라인해서 삭제하는 게 개선.
- NO  → 깊은 모듈이거나, 더 깊게 만들 여지가 있는지 본다.

## 작업 흐름

### 1. 스캔
- `src/` (또는 프로젝트 소스 루트) 전체를 훑는다.
- 다음 패턴 후보로 수집:
  - 한 곳에서만 호출되는 thin wrapper 함수
  - 필드 수와 메서드 수가 거의 같은 클래스
  - getter/setter 만 있는 클래스
  - "Manager", "Helper", "Util" 류 잡동사니
  - 인터페이스 = 구현 1:1 (다형성 없음)
  - 같은 데이터를 여러 모듈이 똑같이 변환

### 2. 후보 목록 제시

각 후보에 대해:

```
### 후보 N: <파일:라인>
- 현재: <한 줄 진단>
- 제안: <인라인 / 합치기 / 깊게 만들기>
- 영향 범위: 호출처 K 곳
- 삭제 테스트 결과: <왜 인라인이/리팩토링이 더 나은지>
- 위험도: 낮음/중간/높음
```

후보는 위험도·가치 기준으로 정렬. 최대 10 개까지.

### 3. 사용자 선택
- "어느 후보를 진행할까요? (번호 입력, 여러 개 가능)"
- 선택된 것만 실행. 나머지는 건드리지 않는다.

### 4. 실행 (선택된 후보당)
- 변경 적용.
- lint && build 통과 확인.
- 테스트가 있으면 실행.
- 한 후보 끝낼 때마다 커밋 (main 직접):
  - 메시지: `refactor: <후보 N> <짧은 설명>`
- 다음 후보로.

### 5. 보고
- 적용한 변경 목록.
- 제거된 LoC, 단순화된 호출처 수.
- 보류한 후보가 있으면 그 사유.

## 안전 규칙

1. **공개 API 시그니처는 사용자 승인 없이 변경 금지.**
2. **테스트 없는 영역은 위험도 "높음" 으로 표시.** 사용자가 알고 동의해야 진행.
3. **한 번에 한 후보씩.** 여러 개 묶어서 커밋하지 않는다.
4. 실패하면 즉시 멈추고 사용자에게 보고. 자동 롤백 시도하지 않는다.
