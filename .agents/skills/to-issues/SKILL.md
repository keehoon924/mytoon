# to-issues

## 목적
PRD 를 "개발자 1 명이 하루~며칠 안에 끝낼 수 있는" 독립 이슈로 쪼갠다.
GitHub 에 실제 이슈로 등록한다.

## 분할 원칙: 수직 슬라이스

레이어별 분할 금지. 기능별 분할.

- ❌ "DB 스키마 만들기" / "API 만들기" / "UI 만들기"
- ✅ "회원가입 (DB + API + UI 전체)" / "로그인 (DB + API + UI 전체)"

각 이슈는 머지되는 순간 사용자가 손으로 만질 수 있는 가치를 준다.

## 작업 흐름

1. **PRD 읽기**
   - 가장 최근 `docs/prd/*.md` 파일 확인.

2. **이슈 후보 추출**
   - 사용자 스토리 단위로 쪼갠다.
   - 너무 크면 다시 쪼갠다 (1 이슈 = 1~3 일).
   - 너무 작으면 합친다.

3. **의존 순서 정렬**
   - 인증 → 핵심 기능 → 부가 기능 순.
   - 다른 이슈를 막는(blocking) 것 먼저.

4. **이슈 본문 작성**

   ```markdown
   ## 목표
   <한 줄 요약>

   ## 수용 기준 (Acceptance Criteria)
   - [ ] ...
   - [ ] ...

   ## 구현 범위
   - DB: ...
   - API: ...
   - UI: ...

   ## 테스트
   - ...

   ## 의존
   - 선행: #<이슈번호> (있으면)

   ## 범위 외
   - ...
   ```

5. **사용자 확인 후 GitHub 등록**
   - 등록 전 전체 목록(제목·요약)을 보여주고 승인 받는다.
   - 승인되면 `mcp__github__issue_write` 로 하나씩 생성.
   - 라벨: `feature`, `bug`, `chore` 중 적절히.

6. **완료 보고**
   - 등록된 이슈 번호 목록 출력.
   - "이슈 등록이 완료됐습니다. 1 번 이슈부터 개발을 시작할까요?" 라고 물어본다.

## 주의

- PRD 에 없는 내용은 이슈로 만들지 않는다.
- "오픈 이슈" 항목은 이슈 등록 전에 사용자에게 다시 확인.
