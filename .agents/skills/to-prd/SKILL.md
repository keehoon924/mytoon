# to-prd

## 목적
지금까지의 대화에서 합의된 내용을 PRD 문서 하나로 굳힌다.
새 질문을 던지지 않는다. 이미 합의된 내용을 정리만 한다.

## 입력
- 직전 /grill-me 세션의 대화 기록
- 코드베이스 현재 상태

## 출력
`docs/prd/<feature-slug>.md` 파일 1개.

## PRD 구조

```markdown
# <기능 이름> PRD

## 1. 문제 정의
- 누가 (사용자/페르소나)
- 어떤 상황에서
- 무엇이 불편한가

## 2. 해결 방안
- 한 줄 요약
- 핵심 아이디어
- 왜 이 방식인가 (대안 검토 포함)

## 3. 사용자 스토리
- US-1: ...로서, ...할 수 있다 (수용 기준 포함)
- US-2: ...
- ...

## 4. 구현 결정
- 스택/라이브러리 선택과 이유
- 데이터 모델 (스키마)
- API 엔드포인트 목록
- UI 화면 흐름
- 인증·권한
- 외부 의존성

## 5. 테스트 결정
- 단위 테스트 범위
- 통합/E2E 테스트 범위
- 수동 QA 시나리오
- 성공 지표 측정 방법

## 6. 범위 외 (Out of Scope)
- 이번에 안 하는 것 명시
- 나중에 다룰 후보

## 7. 오픈 이슈
- 아직 결정 못 한 것 (있으면)
```

## 작성 규칙

1. **대화에서 합의된 내용만 적는다.** 새로 추측해서 채우지 않는다.
2. 빈 칸은 "TBD" 로 두고 끝에 "오픈 이슈"에 모은다.
3. 사용자 스토리는 INVEST 원칙 (독립적·협상가능·가치·추정가능·작게·테스트가능).
4. 구현 결정은 "왜" 를 한 줄이라도 적는다.
5. PRD 작성이 끝나면: "PRD 가 완성됐습니다. /to-issues 로 작업을 쪼갤까요?" 라고 물어본다.
